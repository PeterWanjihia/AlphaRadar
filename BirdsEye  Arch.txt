Good. We keep the product fixed:
AlphaTrace = Trader PNL leaderboard + smart-money movement intelligence.
“Which Solana wallets are actually profitable, and what are those wallets doing now?”
1. Architectural north star
Birdeye raw data → wallet normalization → wallet scoring → wallet ranking → wallet profiling → token-level smart-money aggregation → UI/API presentation.
So the system has two major intelligence layers:
1. Wallet Intelligence Layer
This answers:
“Is this wallet actually good?”
2. Token Signal Layer
This answers:
“What tokens are good wallets touching right now?”
Everything else exists to support those two layers.

2. System boundary
AlphaTrace has a tight boundary.
It does not execute trades.
It does not custody user funds.
It does not connect wallets for signing.
It does not claim to predict price.
It is an onchain analytics and intelligence product.
The system consumes Birdeye data, computes derived metrics, caches results, and exposes trader-facing views.
So the architecture is read-heavy, API-heavy, computation-light, and cache-sensitive.
That matters because our core engineering problems are:
   * API credit management
   * latency
   * data freshness
   * wallet ranking correctness
   * derived-score explainability
   * avoiding fake “smart money” signals
   * presenting complex wallet data simply

The wallet profile is a composed read model.
Flow 3: Build smart-money feed
This is the product’s most important architecture.
Scheduled job / user request
  ↓

  ↓
For each wallet:
    inspect holdings
    inspect balance changes / recent activity
    inspect PNL credibility
  ↓
Group activity by token
  ↓

    count smart wallets involved
    calculate combined Alpha Score of wallets
    fetch token market data
    fetch token security status
    fetch token metadata
    compute Smart Money Signal Score
  ↓

Core endpoints:
   * /v1/wallet/token_list

   * /wallet/v2/pnl

   * /defi/v3/token/meta-data/single or /multiple
The architectural point:
The smart-money feed is an aggregation over wallets, not a token scanner.
That distinction matters.
A token scanner starts from tokens.
AlphaTrace starts from credible wallets, then asks what tokens appear in their behavior.

7. Caching architecture
This project lives or dies on caching.
Birdeye API calls are valuable. Do not waste them.
Use three cache layers.
Layer 1: Request cache
Short-lived memory cache.
TTL:
   * token metadata: 24 hours
   * token market data: 1–5 minutes
   * wallet PNL summary: 10–30 minutes
   * wallet holdings: 2–5 minutes
   * leaderboard: 10–30 minutes
   * smart-money feed: 5–15 minutes
Layer 2: Database snapshots
This is the main cache.
The app can always show the last generated snapshot, even if refresh fails.
That improves reliability.
Layer 3: Background refresh
A cron job refreshes:
   * leaderboard every 15–30 minutes
   * top wallet profiles every 30–60 minutes
   * smart-money feed every 10–15 minutes
   * token metadata lazily
For the bounty, we can run this as a server-side scheduled task or manual refresh endpoint. But architecturally, treat it as a separate worker.

10. Scoring architecture
The scoring should be deterministic, not AI-generated.
That means:
Same input → same score.
10.1 Alpha Score
Alpha Score =
  realized_pnl_score * 0.25
+ roi_score * 0.20
+ win_rate_score * 0.15
+ consistency_score * 0.15
+ wallet_age_score * 0.10
+ recency_score * 0.10
- risk_penalty * 0.05


Each sub-score should be normalized to 0–100.
10.2 Confidence score
Separately compute confidence.
Why?
A wallet with only 2 trades can have 100% win rate. That does not make it elite.
Confidence should depend on:
   * number of trades
   * wallet age
   * number of tokens traded
   * volume
   * data completeness
Output:
alphaScore: 88
confidence: "medium"


This prevents shallow ranking.
10.3 Archetype classifier
Use rule-based classification.
Examples:
Alpha Hunter
   * high ROI
   * strong realized PNL
   * many small/midcap wins
Consistent Compounder
   * moderate ROI
   * high win rate
   * stable net worth curve
One-Hit Wonder
   * high PNL
   * low consistency
   * one token dominates profits
Exit Liquidity
   * negative PNL
   * poor win rate
   * high losses
Inactive Winner
   * good historical PNL
   * low recent activity
This makes the product memorable.
________________


11. Smart-money signal architecture
This deserves special attention.
The feed should not say:
“This token is good.”
It should say:
“This token has smart-wallet attention.”
That is legally and intellectually cleaner.
Token signal computation
For each token found among top wallets:
Signal Score =
  smart_wallet_count_score * 0.25
+ average_alpha_score * 0.20
+ combined_pnl_score * 0.20
+ portfolio_weight_score * 0.10
+ market_liquidity_score * 0.10
+ security_score * 0.10
+ recency_score * 0.05


Token signal explanation
Every signal should include:
   * how many smart wallets are involved
   * average Alpha Score of those wallets
   * combined historical PNL of those wallets
   * whether the token passes basic security checks
   * whether liquidity is acceptable
   * whether the signal is fresh or stale
This is the difference between a toy dashboard and a real analytics product.

13. Error handling architecture
You need graceful degradation.
Possible failures:
   * Birdeye rate limit
   * endpoint timeout
   * wallet has no PNL data
   * token metadata missing
   * malformed wallet address
   * incomplete net worth history
   * security endpoint unavailable
Do not let one missing endpoint kill the whole profile.
Use partial responses:
{
  "data": {...},
  "warnings": [
    "Net worth history unavailable",
    "Token security check unavailable for 2 tokens"
  ],
  "freshness": {
    "pnl": "fresh",
    "holdings": "cached",
    "metadata": "cached"
  }
}


This makes the system robust.
